# CLAUDE.md

A standalone AzerothCore ALE/Lua module providing summonable "premium" NPCs (Profession Trainer, Heirloom Vendor, Class Trainer, Teleporter, plus a gossip-based menu item that summons any of them). No C++ - everything is server-side Lua loaded by ALE.

**Read `README.md` first - it documents the full architecture, access-control layering, reload mechanics, and every current NPC in detail. This file covers conventions and gotchas the README doesn't, not a duplicate of it.**

## Hard invariants

- **Never modify a real, pre-existing row.** Every NPC here borrows a model/displayId or faction from a real creature (e.g. each Profession Trainer entry borrows its profession's own real "Grand Master"-named trainer's model) without ever touching that creature's own `creature_template` row. The Premium Menu item repurposes a confirmed-orphaned item (zero references anywhere in loot/vendor/quest tables) rather than creating a new `item_template` row. Before reusing *any* existing entry, confirm it's either read-only-borrowed (model/displayId only) or genuinely orphaned (query for references first) - don't assume.
- **Never duplicate the access-check-then-summon logic.** Every caller (dot-command, the menu item) goes through that NPC type's own shared `Try<Name>(player)` function (`TryProfessionTrainer`, `TryHeirloomVendor`, etc.), which itself calls `IsPremiumNpcAllowed`. If you're writing a new entry point for an existing NPC type, call the existing `Try<Name>` - don't re-implement the check.
- **Custom `creature_template` entries use the `9002xx` range** (900200-900226 so far, contiguous - 900200 itself is retired and no longer used by any creature, see the Profession Trainer redesign in README.md; next new one is 900227). Keep using this range for consistency, and remember new entries need a full worldserver restart to be picked up (not `.reload ale`, not `.reload creature_template`).

## Gotchas already hit and fixed - don't re-discover these

- **A gossip menu with exactly one option, no popup string, and no code, gets auto-selected by the client instead of ever being shown.** Confirmed via tracing the actual client/server gossip flow, not assumed. Any new gossip menu in this project needs to guarantee 2+ options (or a popup confirmation) on every code path, even ones that seem like they'd "obviously" only ever have one entry (e.g. a filtered list down to a single allowed item) - this has bitten this project before (see `premium_menu_item.lua`'s filtering logic for the pattern that avoids it).
- **Return-value semantics differ between creature and item gossip hooks, and they're opposite.** For `RegisterCreatureGossipEvent` hooks, returning `true` suppresses the native `PrepareGossipMenu`/`SendPreparedGossip` flow - you only return `true` once you've sent your own menu. For `RegisterItemGossipEvent` hooks, returning `false` suppresses the item's attached on-use spell from casting (used in `premium_menu_item.lua` to stop its required placeholder spell from actually firing). Don't assume one convention applies to both.
- **The native Trainer window only ever sends its spell list once per "session"; it does not reliably redraw colors after you learn something mid-session**, even though the server-side state (`Trainer::GetSpellState`) is recomputed correctly every time and the actual purchase gate is never wrong. `profession_trainer.lua`'s `OnPlayerLearnSpell` handler works around this by forcing the window to close (an immediate `SendTrainerList` resend, no delay) rather than trying to refresh it in place - a delayed resend was tried first and silently did nothing (client ignored the second list), so don't re-attempt an in-place refresh without revisiting that history first.
- **A single creature's `npc_vendor` list silently truncates at 255 total rows, well below the 150-item display cap, and nothing logs an error.** `VendorItemData::GetItemCount()` (core's `CreatureData.h`) returns `uint8`; `m_items.size()` (a real, unbounded `std::vector`) gets truncated mod 256 the moment it's returned, and `SendListInventory`'s loop only ever inspects that truncated number of slots, in the order they were inserted. Glyph Vendor first shipped as one shared `creature_template` entry holding all ten classes' glyphs (353) filtered per-player via the `conditions` table (`CONDITION_SOURCE_TYPE_NPC_VENDOR` + `CONDITION_CLASS`) - which correctly keeps every player's *filtered* view under the 150-item display cap (`ItemHandler.cpp`'s `MAX_VENDOR_ITEMS`), but that filtering runs *after* the uint8 truncation already cut the list down to the first 97 (353 mod 256) entries by insertion order, regardless of class. Some classes' glyphs landed outside that truncated range and got an empty vendor window - confirmed live, not caught by checking `MAX_VENDOR_ITEMS` alone. Fixed by splitting into one `creature_template` entry per class (`sql/db-world/07_glyph_vendor.sql`, `PREMIUM_NPC_CONFIG.GLYPH_VENDOR.ENTRIES`) so no single entry's `npc_vendor` list ever approaches 255. **Before attaching more than ~200 `npc_vendor` rows to one creature for any future NPC here, split it instead** - conditions-table filtering does not help, since the cap is hit before filtering ever runs.
- **A real creature's name implying full coverage (e.g. "Grand Master X Trainer") doesn't mean its own `trainer_spell` list actually spans that profession's whole skill range, and on this server that data's provenance can't always be verified.** Checked directly against this server's `trainer_spell` table, not assumed: most profession "Grand Master" trainers' own rows do span their entire 0/1-to-max chain, but at least one (Alchemy) doesn't, and the full-range ones didn't match Wowhead's retail reference either (this server runs `mod-individual-progression`, which patches a lot of trainer data - confirmed it doesn't *dynamically* rewrite `trainer_spell` per player phase, but couldn't conclusively trace whether it or stock AzerothCore data explains the mismatch). Profession Trainer (see README.md) ended up giving every profession its own small custom `trainer_spell` entry instead of reusing any real trainer creature directly, specifically to avoid depending on real data of unverifiable origin. Before pointing any future NPC here at a real trainer/vendor entry by name/reputation alone, compare its actual data against the full real chain *and* sanity-check it against an external reference if one's available - don't trust this server's live data as automatically representative of "how the real game works," even though it's what actually governs runtime behavior here.
- **A `creature_template` row with `AIName = 'SmartAI'` can override the follow movement this module relies on.** One candidate real Grand Master First Aid trainer (entry 23734) had a `smart_scripts` "On Spawn - Start WP" entry that would make a summoned copy walk a waypoint path instead of following the player - caught by checking `smart_scripts` for the entry before reusing it, not by symptom. Before reusing any real creature entry for a new premium NPC, check its `AIName`/`ScriptName` and (if `SmartAI`) its `smart_scripts` rows first; if it has movement/aggro/dialogue scripting, pick a different clone of the same trainer/vendor instead - most "Grand Master"-style trainers have several interchangeable clones with identical data, just different names/locations (e.g. Class Trainer's faction-specific `ENTRIES`, or Glyph Vendor's per-class `ENTRIES`).
- **`GetActivePremiumNpcGuid` (`premium_npc_summon.lua`) only tracks who *summoned* an NPC, not who's currently using it.** This module deliberately lets anyone nearby use an already-summoned NPC (see the access-scope note in README.md), so a second player interacting with someone else's summoned vendor/trainer has a `nil` `GetActivePremiumNpcGuid(player)`. Don't use that function to gate any future per-buyer/per-user logic keyed to interaction with a specific NPC instance (e.g. "do something when *this player* buys from *this* vendor") - it will silently miss everyone except the summoner. (Also worth knowing if such logic is ever needed: ALE has no vendor-purchase-specific hook at all - `PLAYER_EVENT_ON_STORE_NEW_ITEM` fires for every way an item lands in inventory, not just vendor buys - so detecting a purchase precisely means going lower, e.g. reading the buyer's own `CMSG_BUY_ITEM`/`CMSG_BUY_ITEM_IN_SLOT` packet via `PACKET_EVENT_ON_PACKET_RECEIVE`.)

## Verify against current source, don't guess

This project's reliability has come from checking real source/data before relying on it, not from general WoW knowledge - aura/spell IDs, gossip semantics, icon names, and reload behavior have all had at least one "looked right but was wrong" moment this way. Before relying on any spell/aura ID, icon name, or hook signature: check it against the installed `acore_world` database or `ScriptMgr.h`/ALE's `LuaEngine/methods/` directly rather than trusting memory or an external reference. If something can't be verified (e.g. no `SpellIcon.dbc` mirror table exists in this database, so icon *names* can't be cross-checked via SQL), say so explicitly rather than asserting confidence that isn't backed by a real check.
